# Workspace Snapshot (Export/Import)

## Function
Serializes the live workspace — sessions (title, SSH config, startup command, accent, model, room assignment), project sub-tabs, and session metadata — to a JSON snapshot that can be saved to the server and reloaded later.

## Purpose
So users can recreate their multi-session layout after a server restart, across machines, or after the desktop app relaunches. Auto-saves on change, auto-loads on boot.

## Implementation
- **Storage path** (`WORKSPACE_SNAPSHOT_PATH`, `server/apiRouter.ts`): `process.env.APP_USER_DATA/workspace-snapshot.json` (Electron) or `<server-dir>/../data/workspace-snapshot.json` (Node server). Single slot — no multi-slot support. Written with `JSON.stringify(..., null, 2)`.
- **Save dedup (8-field key)**: server rejects duplicates keyed on `(title, sshConfig.host, sshConfig.port, sshConfig.username, sshConfig.workingDir, sshConfig.command, startupCommand, originalSessionId)` joined with `\0`. The 8th field (`originalSessionId`) was added to prevent collapsing legitimately distinct sessions that share the same workingDir + title (e.g., multiple Claude conversations against the same project). Client-side `sessionDedupeKey()` in `src/lib/workspaceSnapshot.ts` uses the matching 8-field formula. Logs how many duplicates were removed.
- **Save validation**: rejects missing `version` or non-array `sessions` with 400.
- **Load**: 404 when snapshot absent — client treats as empty workspace.
- **Snapshot shape**: top-level `WorkspaceSnapshot = {version: 1, exportedAt: number, sessions: SessionSnapshot[], rooms: Room[]}`. Each `SessionSnapshot` carries `originalSessionId` for remapping room assignments on import, plus: `title`, `status`, `accentColor`, `characterModel`, `pinned`, `muted`, `alerted`, `enableOpsTerminal`, `sshConfig`, `startupCommand`, `permissionMode`, `projectTabs`, `fileTabs`, `queueItems`, `isFork`, `originSessionId`.
- **Export gating** (`buildSnapshot`): a session is exported only if it has an `sshConfig` AND is not `archived` AND `status !== 'ended'` (ended/killed/archived sessions can't be meaningfully restored). `snapshotRooms` is filtered so each room's `sessionIds` references only exported session ids. `enableOpsTerminal` is set true when the session has (or had) an ops terminal.
- **`queueItems` carries full automation** (`buildSnapshot` capture / `importSnapshot` restore): each entry stores `text, position, createdAt, images, type, intervalMs, runAt, beforeChain, afterChain, excludeWindows, firstFireOfDay, disabled, totalFires` (typed as `Pick<QueueItem, …>`). **Regression fixed (Jun 2026):** previously only `{text, position, createdAt, images}` were captured/restored, so a restored **loop** came back with no `type` → it defaulted to `'once'` → the scheduler consumed/deleted it on its first fire and the loop silently vanished from the queue panel after every reopen / Restore-Workspace. On restore, timing is re-armed from now: `loop` → `now + intervalMs` (falling back to `60_000` when `intervalMs` is absent/≤0); `schedule` → its absolute `runAt` (or `now + 60s` if absent); `once` → `nextFireAt: 0`. `execState`/`execStepIdx` are intentionally NOT restored (a chain can't safely resume into a freshly recreated terminal, so items come back idle). Only sessions with `sshConfig` are exported, so this path applies to SSH sessions; local-session queues persist independently via the IndexedDB `promptQueue` table keyed by `sessionId`. The scheduler semantics that consume these fields live in [Queue Scheduler](./queue-scheduler.md).
- **Project sub-tabs**: `ProjectSubTab` preserves per-session file-browser tabs (path, label, `customLabel`, `initialPath`, `initialIsFile`).
- **Auto-save debounce**: `useWorkspaceAutoSave` subscribes to `sessionStore.sessions` + `roomStore.rooms` and calls `scheduleAutoSave()` on change, which coalesces rapid changes into one save after `AUTO_SAVE_DEBOUNCE_MS = 5_000`. Auto-save is gated three ways: it no-ops while `_restorePending` (default `true` until auto-load resolves the Restore dialog — prevents a first save overwriting the snapshot), while `_importInProgress`, and when no session has an `sshConfig`. `flushSave()` skips the debounce for explicit saves (app quit) and honors the same `_restorePending` guard.
- **Dynamic import**: `App.tsx` (`await import('@/lib/workspaceSnapshot')`, ~line 93) imports `flushSave` lazily on quit to avoid pulling the module into the initial bundle.
- **Selective restore on startup** (`RestorePickerModal`): on every AASC launch, `useWorkspaceAutoLoad` calls `requestRestoreSelection(snapshot)` which opens the picker modal. The modal groups snapshot sessions by their room membership (snapshot rooms + any extra rooms in `localStorage['session-rooms']`), shows per-session warnings (`⚠ new` for non-UUID `originalSessionId`, `⚠ fork` for Claude `--fork-session` or Codex `fork` commands), and resolves a `PickerResult { selectedIds: Set<string> | null, cancelled: boolean }`. `importSnapshot` accepts an optional 3rd argument `sessionFilter: Set<string> | null` and filters `dedupedSessions` by `originalSessionId` after dedup but before launch — room remap is unaffected (unselected sessions naturally drop out of `idRemap` and the orphan bucket). The "Don't ask again — auto-resume on every restart" checkbox writes `restore-picker:auto-resume-all=1` to localStorage; subsequent restarts skip the modal entirely. The last selection is persisted under `restore-picker:last-selection` and intersected with the current snapshot on next open (falls back to all-selected if intersection is empty). `Escape` = Resume nothing (cancel, no overwrite); `⌘/Ctrl+Enter` = Resume selected. Cancel returns early without launching anything, preserving the snapshot for the next restart.
- **Auto-resume on import**: when `originalSessionId` looks like a real CLI session UUID (matches `^[a-zA-Z0-9_-]+$` and does NOT start with `term-`), the client passes it as `resumeSessionId` to `POST /api/terminals`. The server rebuilds the launch command by CLI: Claude strips old `--resume`/`--continue`/`--fork-session` flags, preserves permission mode flags, and runs `<baseCmd> --resume '<UUID>' || <baseCmd> --continue`; Codex strips old `resume`/`fork` subcommands and runs `<baseCmd> resume '<UUID>' || <baseCmd> resume --last`. Synthetic `term-*` IDs use the fallback (`claude --continue` or `codex resume --last`) where applicable. The terminal is spawned with an empty command (so `createTerminal` skips auto-launch) and the rebuilt command is injected via `writeWhenReady`. This ensures re-imports pick up the user's actual conversation instead of re-forking from the ancestor or starting fresh. Other non-Claude/Codex commands (for example Gemini) run the saved command verbatim.
- **`term-*` command rerouting**: For synthetic `term-…` sessions (no real Claude UUID), the client now sends `command: ''` and routes the original `cfg.command` through `startupCommand` (which has no shell-metacharacter validator on the server). This allows compound commands like `npm run dev && claude` to import without being rejected by `validateCommand` in `sshManager.ts`. Existing snapshot `startupCommand` is preferred over `cfg.command` when both exist.
- **`suppressBroadcast` clear-all**: `importSnapshot` calls `POST /api/sessions/clear-all` with body `{ "suppressBroadcast": true }`. The server then calls `clearAllSessions()` without broadcasting `CLEAR_BROWSER_DB` over WebSocket, eliminating the race where the broadcast wipes IndexedDB entries for newly created sessions mid-import. The Reset button (manual user action) still sends no body and gets the broadcast.
- **Orphan session bucketing**: After room remap, any session whose new ID isn't referenced by any room's `sessionIds` is bucketed into a synthesized `Ungrouped` room (or appended to an existing one named `Ungrouped`). This prevents sessions that exist in `snapshot.sessions[]` but aren't listed in any `rooms[].sessionIds` from becoming invisible after import. Orphans are NOT auto-matched to rooms by name — bucketing is conservative.
- **Failure surfacing**: `importSnapshot()` returns `{ created, failed, failedTitles[] }` (the third field is new). `useWorkspaceAutoLoad` passes `failedTitles` to `WorkspaceLoadingOverlay`, which switches to an error panel listing each failed title with a `DISMISS` button. The overlay does NOT auto-hide while errors are visible. On full success it dismisses normally.
- **`workingDir` existence fallback** (`server/sshManager.ts`): `createTerminal` checks `existsSync(workDir)` before `pty.spawn`. If the directory no longer exists (e.g., user deleted the project), the spawn falls back to `os.homedir()` with a `log.warn` instead of throwing ENOENT. The session card is still created so the user can re-cd manually.
- **Per-workDir FIFO `pendingLinks`** (`server/sshManager.ts`): `pendingLinks` was promoted from `Map<string, PendingLink>` to `Map<string, PendingLink[]>` (FIFO queue). Multiple sessions sharing the same `workingDir` (e.g., 6 thesis sessions in `/Users/.../thesis`) now each get a distinct slot — the previous last-write-wins behavior caused all but the last to fail to bind their `SessionStart` hook to the correct terminal card. `consumePendingLink(workDir, terminalId?)` accepts an optional 2nd argument for surgical removal; default behavior pops the front.
- **Scrollback restoration on import**: `POST /api/sessions/clear-all` returns `{ savedOutputs: [{ key, data }] }` where each `key` is `title\0workDir` and `data` is base64. `importSnapshot` builds a `savedOutputMap` from these, then after each session is recreated POSTs the matching buffer to `POST /api/terminals/:id/prefill-output` so xterm scrollback survives the export → import roundtrip.
- **In-flight import gate**: module-level `_importInProgress = true` is set at the top of `importSnapshot()` and `cancelAutoSave()` is called immediately. `scheduleAutoSave`'s debounced callback bails when `_importInProgress` is true (and other callers can query `isImportInProgress()`). After `onComplete`, a `POST_IMPORT_QUIET_MS = 3_000` `setTimeout` clears the flag once in-flight Zustand/WS updates settle, preventing auto-save races that could re-publish stale mid-import state (old dying ids mixed with new ids).
- **Pre-creation metadata bundling**: `pinned`, `muted`, `alerted`, `accentColor`, `characterModel` (plus `isFork`/`originSessionId`) are sent in the initial `POST /api/terminals` body instead of as fire-and-forget `PUT` calls after creation. This eliminates a class of race conditions where the metadata patch could land before the session row was fully visible.
- **Reconnect-terminal trigger**: when the server returns `{ deduplicated: true, hasTerminal: false }` (an existing session was matched but its PTY is gone), the client POSTs `/api/sessions/:id/reconnect-terminal` to respawn the terminal under the existing session id rather than creating a new one.
- **Room merge**: existing rooms in `localStorage['session-rooms']` not present in the snapshot (matched by room `id`) are merged with the snapshot's `rooms` before remap, then all `sessionIds` are filtered through `idRemap` (and `validNewIds`) so the user's pre-import room layout is preserved alongside imported rooms with no stale ids.
- **`canonicalMap` for dropped duplicates**: when import-time dedup drops a snapshot session because an earlier session shares its 8-field key, the dropped session's `originalSessionId` is recorded in `canonicalMap` (droppedId → canonicalId) and later folded into `idRemap` to point at the canonical session's new terminal id. This preserves room bindings for sessions that referenced the dropped duplicate by `originalSessionId`.
- **Re-keyed session reconciliation**: after rooms are remapped, sessions whose CLI `SessionStart` hook has already re-keyed `terminalId → hookSessionId` are reconciled — for any session where `sessionId !== terminalId` and the new id isn't in a room but the old `terminalId` is, `roomStore.migrateSession(terminalId, sessionId)` moves the room binding onto the live id.
- **Floating popup (fork) restore**: sessions are skipped only when they have no `sshConfig` (`buildSnapshot`: `if (!session.sshConfig) continue`). A floating Explain/Translate popup that DOES carry an `sshConfig` is exported with `isFork` + `originSessionId` set, and re-created on import. Float re-open happens in a **second pass** (`pendingFloatReopens`) AFTER `idRemap` is fully built, so each popup re-links to its origin session's NEW terminal id (re-opening inline would pin it to the stale snapshot id, possibly before its origin is processed, leaving it invisible). On the create side, popups whose origin session is NOT part of this restore are dropped first (so their PTY is never spawned as an invisible orphan); in the second pass any popup whose origin failed to launch is also skipped. See [Floating Terminal Fork](./floating-terminal-fork.md). Popups with no `sshConfig` (the common case) are still never exported.

### Pinned sessions — "always-on" recreate

A session pinned from the sidebar (`RobotListSidebar` pin toggle → `sessionStore.togglePin` → `PUT /api/sessions/:id/pinned`, persisted in the snapshot's `pinned` field) is treated as **"always there"**:

- **Sidebar**: pinned sessions sort to the **top of their project group** (`src/lib/sessionSort.ts`) with a yellow left-accent bar + filled pin icon.
- **Auto-restore on restart** (`useWorkspaceAutoLoad`): pinned `originalSessionId`s are unioned into the restore filter **regardless of the picker outcome** — they restore even if the user cancels, and the picker is skipped entirely when every snapshot session is pinned. A pin therefore overrides an unchecked pinned row; to stop a pinned session restoring, unpin it.
- **Live respawn** (`src/lib/pinnedRespawn.ts`): when a pinned session's process dies (a fresh `status → 'ended'` transition seen in `useWebSocket`'s `session_update`), `onSessionEnded` relaunches it with the same config (`buildRespawnBody` mirrors `importSnapshot`'s per-session create body — real CLI ids resume via `resumeSessionId`, `term-*` raw commands route through `startupCommand`). Backoff 2s→4s→8s; a crash-loop cap (3 deaths / 60s, keyed by the **stable** `projectPath+title` so it survives the id change a respawn causes) gives up with a toast.
- **Deliberate close vs. crash**: closing a pinned session from the sidebar `✕` confirms, then **unpins** it (so it won't auto-recreate) AND calls `markUserClosing(session)` (belt-and-suspenders against the async unpin race), so its imminent `ended` event — and any already-scheduled respawn — is cancelled. Only an *unexpected* death of a *still-pinned* session respawns.

## Change Risks
- Changing `SessionSnapshot` shape without a version bump silently breaks `/api/workspace/load` for old snapshots
- The dedup key formula must stay synchronized between client (`src/lib/workspaceSnapshot.ts:sessionDedupeKey`) and server (`server/apiRouter.ts` save handler). Both currently use the 8-field formula `[title, host, port, user, workdir, command, startupCommand, originalSessionId]` joined with `\0`. If you reorder, rename, or add fields, update both files in the same commit or sessions will be silently dropped from one side
- Session ID remapping must run before room reconciliation — otherwise rooms attach to wrong sessions
- The resume command stripping helpers in `apiRouter.ts` must keep parity between the `/sessions/:id/resume` flow and workspace auto-load. Adding new Claude flags or Codex resume/fork subcommands requires updating the centralized builders, not just one endpoint.
- `resumeSessionId` passes through Zod as alphanumeric+dashes/underscores only — any Claude or Codex session ID format change (e.g., adding dots/colons) requires updating the regex in both `terminalCreateSchema` and the client-side `looksLikeRealSessionId` check
- `suppressBroadcast` is a soft contract — if a future caller of `/api/sessions/clear-all` needs the broadcast suppressed during a multi-session orchestration, send `{ suppressBroadcast: true }` and remember to manually reset client state. The Reset button MUST continue to send no body so it broadcasts normally
- `pendingLinks` is now an array-per-workDir queue. Any new caller must use the exported helpers (`addPendingLink`, `tryLinkByWorkDir`, `consumePendingLink`) — direct `Map.set/get` access on `pendingLinks` will break when multiple sessions share a workDir
- `term-*` synthetic IDs route the user's command through `startupCommand` to bypass `validateCommand`'s shell-metachar regex. If `startupCommand` ever gains its own validator, the `term-*` import path will need a new escape route (e.g., `bash -c` wrapping). Currently safe because `startupCommand` is just `writeWhenReady` to the PTY shell.
- `workingDir` fallback to `os.homedir()` is silent — only a `log.warn`. If users complain about sessions opening in `~` instead of their project, the fallback was triggered. Consider surfacing this in the failure overlay if it becomes common.
- `failedTitles` lives in a module-level publisher (`reportWorkspaceLoadErrors` in `workspaceSnapshot.ts`). If you replace the overlay with a Zustand store, migrate the publisher or the failure list will silently disappear from the UI
