# Prompt Queue System

## Function
Per-session prompt queuing with drag-and-drop reordering, cross-session moves, image attachments, optional auto-send on waiting/input status, and a global favorites history that lets users reuse a saved queue item (Once / Loop / Schedule) in any session.

## Purpose
Users can queue up multiple prompts for a session and have them sent automatically, enabling batch workflows. The global queue history lets a queued item (or full chain) be favorited once and re-applied to fresh sessions — useful when a "review + lint + test" loop pattern is repeated per project.

## Implementation
- QueueItem: {id, sessionId, text, position, createdAt, images?: QueueImageAttachment[]}
- QueueImageAttachment: {name, dataUrl} — up to 5 attachments per item. Three input methods:
  - **File picker** (paperclip button): accepts images + common text files (.pdf, .txt, .md, .json, .csv, .xml, .yaml, .log)
  - **Clipboard paste** (Cmd/Ctrl+V): captures pasted images/files from clipboard via `onPaste` handler on the compose textarea. Binary data is intercepted (prevents pasting as garbled text)
  - **Drag and drop**: drop files onto the compose area; cyan border highlights the drop zone during drag-over
- Queue operations: add (append), remove (by id), reorder (drag-and-drop), moveToSession (dropdown picker), send now, edit (inline textarea)
- Auto-send: **per-session** toggle (the ➤ green paper-plane icon in the queue header), stored in each session's `QueueAutomationConfig.autoSend` and persisted per session to the `queueAutomation` IndexedDB table (defaults to ON). Toggling it on session A never affects session B. When a session transitions to 'waiting' or 'input' status, its first queued item is sent to the terminal via POST /api/terminals/{terminalId}/write, only removed after a successful send. **Both** QueueTab mounts for that session AND `useGlobalQueueScheduler` read the SAME per-session value (`automationConfig.autoSend`, re-read each tick), so the visible toggle and the actual firing can never disagree — `QueueTab` is mounted twice (always-on strip in `DetailPanel` + the Queue tab), and both pass the same `sessionId`. Older `queueAutomation` rows saved before auto-send became per-session lack the `autoSend` column and are read as ON (back-compat default). Note: `settingsStore.autoSendQueue` is an unrelated, currently-unused settings flag (consumed only by the settings UI) — it does NOT gate firing; the per-session `autoSend` is the real control.
- Auto-Enter: **per-session** toggle (the ↵ cyan return-arrow icon, left of the ➤ paper-plane), stored in `QueueAutomationConfig.autoEnter` and persisted to the same `queueAutomation` table (defaults to ON). Controls whether the send appends a real Enter keystroke (`\r`) to actually submit the prompt in Claude Code / Codex / Gemini TUIs. When OFF, the text is typed into the input box only and the user presses Enter themselves. auto-send governs *when* the prompt leaves the queue, auto-enter governs *how* it is delivered, and the scheduler reads `automationConfig.autoEnter` at fire time so toggles take effect on the next tick. **Invariant — Auto-Enter ON ⟹ Auto-send ON:** `setAutoEnter(id, true)` also flips `autoSend` ON in the same store update, because "Auto-Enter on" must mean the prompt is actually sent *and* submitted (the old decoupling silently produced "typed but never fired" when a user enabled Auto-Enter while Auto-send was OFF). Disabling Auto-Enter leaves Auto-send untouched (auto-fire + typed-only is still valid). The same invariant is **self-healed on load**: `loadFromDb` coerces any persisted `autoEnter && !autoSend` row to `autoSend: true`, so existing broken configs fix themselves on next launch. (History: auto-send/auto-enter were global `localStorage['queue-auto-send'|'queue-auto-enter']` flags through early Jun 2026, then made per-session so each agent can run its own automation policy; the Auto-Enter⟹Auto-send coupling was added shortly after.)
- Send mechanism: POST /api/terminals/{terminalId}/write with {data: text + image paths + terminator}, where terminator = `\r` when auto-enter is ON, otherwise `''`. **Note:** `\n` would only insert a newline inside the TUI input box without submitting; `\r` mimics a real Enter keypress. Images uploaded first via POST /api/queue-images
- Global QueueView: shows all sessions' queues grouped by session, add prompt to any session via session selector (Select dropdown), table layout with MOVE/DEL actions
- Persistence: IndexedDB promptQueue table via Zustand subscribe callback, survives page reloads. Images serialized as JSON in DbQueueItem.images field
- **Loop durability (a loop must never silently vanish)**: `advanceAfterFire` (`src/lib/queueScheduler.ts`) treats a `loop` with a missing/zero/NaN `intervalMs` as a **clamp-and-reschedule** (60s default) and *heals* the stored `intervalMs` in the returned patch — it does NOT remove the item. Previously such a loop was deleted (`{action:'remove'}`) which is exactly the "my loop disappeared after one fire" symptom. Belt-and-suspenders against bad intervals entering the system: `queueHistoryStore.snapshotItem` clamps a loop's interval to 60s when saving a favorite, and `queueHistoryExport.coerceEntry` sanitizes (not rejects) an imported loop entry whose `intervalMs` is missing/≤0. The most common upstream source of a typeless/intervalless loop was the workspace-snapshot restore (see [Workspace Snapshot](./workspace-snapshot.md) — `queueItems` now carries the full automation). **Resume re-key race — fixed (Jun 2026):** on `claude --resume` a re-key (`migrateSession`) used to orphan a loop that was still in IndexedDB-but-not-yet-hydrated when the `session_update` arrived before `queueStore.loadFromDb()` finished. `src/main.tsx` now `await`s `loadFromDb()` (queue + history) **before** rendering `<App>` (which mounts the WebSocket), making the order deterministic — load → render → WS connect → session_update → `migrateSession` (now always sees hydrated items, so it re-keys instead of no-oping).
- Session ID migration: queueStore.migrateSession re-keys all items when a session is replaced (e.g., claude --resume)
- QueueTab collapsible: localStorage['queue-panel-collapsed'], defaults to collapsed
- **Global queue history (favorites)**: top-right 📚 button in the QUEUE header opens `QueueHistorySheet`. Each queue row carries a ★ button (`queueFavBtn`); clicking saves a snapshot to Dexie `queueHistory` (v5 schema) and stamps `QueueItem.historyId` so the star renders filled. Clicking the filled star silently removes the saved entry. History entries store a deep-cloned `QueueItem` with session-local fields stripped (id / sessionId / position / nextFireAt / execState / totalFires) — the snapshot is portable across sessions
- **Apply flow**: in the sheet, [+ Apply] clones the saved snapshot into the currently viewed session (target is implicit, no picker). Loop `nextFireAt` is recomputed as `now + intervalMs`; `historyId` is set on the new live item so the ★ stays filled; `usedCount` and `lastUsedAt` increment on the history entry
- **Edit / View flow**: [✎ Edit] opens `QueueItemEditModal` with `title="Edit history entry"` — Save writes to `queueHistory.update`, NOT to any session's queue. [👁 View] opens a read-only `QueueHistoryViewModal` (same component file) showing name (alias) / type / interval / runAt / chains / source breadcrumb / usage stats
- **Entry alias (display name)**: each history row carries an optional user-chosen `alias` (`QueueHistoryEntry.alias`, persisted to the non-indexed `DbQueueHistory.alias` column — no Dexie version bump). The row shows the alias as a clickable cyan title, or a muted "+ Add name" affordance when unset; clicking turns it into an inline `<input>` (Enter / blur saves via `queueHistoryStore.setAlias`, Esc cancels, empty clears back to null). The alias is searched by the sheet's filter alongside text + source title, surfaced in the View modal as a "Name" field, and round-tripped through export / import (`ExportedEntry.alias`, blank/whitespace coerced to none). It is a label only — it does NOT affect Apply (which clones the snapshot `item`, not the alias)
- **History coupling to queue store**: when a history entry is removed (via 🗑 in the sheet, or by toggling the filled star), `removeEntry` walks every session's queue and clears `historyId` from any live row pointing at the deleted entry — prevents orphan filled stars
- **Per-item on/off toggle**: each queue row has a power-icon button at the leftmost edge. Clicking sets `QueueItem.disabled = true`; the row dims (~55% opacity) with a vertical pause-stripe on the left, and the meta line replaces the next-fire countdown with `— paused —`. Scheduler integration: `pickNext` and `advanceBlockedLoops` (in `src/lib/queueScheduler.ts`) filter out `disabled` items up-front, so loops don't tick, schedules don't fire at their `runAt`, and once items stay queued. Re-enabling a loop resets `nextFireAt = now + intervalMs` to avoid an immediate stale fire from a frozen-in-the-past `nextFireAt`. Status row appends `(N paused)` so the user can see at a glance that some items are inert. `disabled` is stripped from history snapshots, so a freshly applied entry always comes in enabled
- **Chain step gating (saw-work gate)**: a `before → main → after` chain advances one step per fire, but each step must wait for the PREVIOUS step's turn to actually FINISH before the next is sent. Without this, the after/post-chain prompt (and even the next loop cycle) gets typed on top of the still-running prompt and the input box piles up with un-submitted commands. The gate lives in `useGlobalQueueScheduler`'s `chainGateRefs: Map<sessionId, {itemId, sawWork, openedAt}>` (same pattern as `firingRefs`/`coolDownRefs`). After a `continue` fire it opens a gate; each tick the hook records `sawWork=true` when the session is observed busy (`!isSendableStatus` → prompting/working/approval; runs BEFORE the idle-guard/skip-prompting early-returns so a busy tick is never missed); the pure helper `chainGateDecision()` (in `queueScheduler.ts`) then decides hold/fire.
  - **Completion signal = `'waiting'` (Stop), NOT `idle`.** Once `sawWork` is true, the gate releases ONLY when `status === 'waiting'` (the genuine Stop / turn-complete signal). It deliberately ignores decayed `idle`: `AUTO_IDLE_TIMEOUTS.working = 180_000` flips a genuinely-busy session `working → idle` after 3 min of hook silence, so a long task that thinks quietly would otherwise look "done" and the chain would barge in. `input` also holds (the running agent is asking the user, not finished). In the `sawWork` branch there is NO time fallback — it waits strictly for Stop.
  - **No-op fallback (sawWork=false only)**: a step that never visibly goes busy (local slash commands like `/clear` that emit no Stop) releases after `NO_WORK_FALLBACK_MS = 12_000` of continuous sendability, so a mid-chain no-op can't stall the chain forever. This fallback can never fire once a real turn has been observed.
  - Applies to every chain transition (before→before, before→main, main→after, after→after) and to loops + schedules alike; works whether `idleGuard` is ON or OFF. Because the chain never *completes* prematurely it stays in-flight (scheduler PRIORITY 0), which transitively prevents the NEXT loop cycle from starting while the current one is still running. Gate is cleared on `remove`/`reschedule` and when a fresh (non-executing) item is picked
- **HOLD-within-cycle vs SKIP-across-cycles**: the two timing rules operate at different boundaries. (a) *Within* the in-flight cycle, the remaining steps are HELD — they belong to the running cycle and must still execute (the saw-work gate above defers them step-by-step, never drops them). (b) *Across* cycles, a NEW loop cycle that comes due while ANY chain is mid-flight in the same session is SKIPPED (dropped), not deferred — the user never wants a stale cycle to fire the instant the session frees up. Implemented in `useGlobalQueueScheduler`: each tick, if `items.some(isExecuting)` (a chain is in-flight), `advanceBlockedLoops(items, now)` rolls every OTHER due loop's `nextFireAt` forward by one interval. `advanceBlockedLoops` excludes the executing item, so the in-flight cycle's own cadence is untouched (it reschedules normally on completion). Single-loop sessions are unaffected (no other loops to skip) — their pile-up is already prevented by the gate. Schedule items are NOT skipped (they keep their explicit `runAt` and fire when free, by design)
- **Manual ⚡ NOW runs the full chain (force-start)**: clicking the ⚡ NOW button on a loop/schedule item (`QueueTab.handleTriggerNow`) no longer sends just the main prompt — it runs the item's complete `before → main → after` chain in sequence. It does this by setting a transient `QueueItem.forceStart` flag and clearing `execState`; the **global scheduler** then drives the chain with the SAME saw-work gate as an automated cycle (each step waits for the prior step's Stop before the next is typed). `pickNext` gives a fresh `forceStart` item a dedicated **PRIORITY 0.5** (above once/loop/schedule, below an in-flight chain — so two chains never interleave) and lets it bypass idle-guard, quiet-hours, the daily-start clamp, AND skip-prompting: a manual trigger is a deliberate user action, so step 1 fires immediately even mid-work. The bypass is one-shot — `advanceAfterFire` clears `forceStart` on the first advance (both the `continue` patch and the reschedule `completePatch`), so steps 2…N are paced normally by the gate. On completion a loop reschedules (`nextFireAt = now + interval`, cadence restarts from the trigger point) and a schedule is removed — identical to a normal cycle's outcome. Items with **no** chain behave exactly as before (main prompt only, then reschedule/remove). **Works with auto-send OFF**: the scheduler's blanket `if (!autoSend) return` was replaced by a per-session rule — when auto-send is OFF the scheduler still (a) starts a `forceStart` item and (b) drives any in-flight chain to completion (chains are atomic), and does nothing else (no cadence churn, no auto-fires). `forceStart` is **transient / in-memory only** — it is deliberately absent from all three field-whitelist persistence paths (`persistSessionQueue` → IndexedDB, `snapshotItem` → queue history, `buildSnapshot` → workspace snapshot), so it can never survive a reload and re-fire a stale chain. **Guards (both the ⚡ button `disabled` state AND a `handleTriggerNow` early-return, belt-and-braces):** the trigger is blocked when the item is paused (`item.disabled`), when its chain is **already mid-flight** (`isExecuting` — ⚡ NOW only ever STARTS a fresh chain; restarting a running one would reset `execState`, delete the saw-work gate, and type step 1 on top of the still-working agent, breaking chain atomicity), and when the **session's automation is Paused** (`automationConfig.paused`, a hard stop the scheduler honors at the top of `evaluateSession` before any force path — so the trigger must honor it too instead of leaving `forceStart` set to fire unexpectedly on Resume). Trade-off: step 1 fires on the next scheduler tick (≤1s) rather than perfectly synchronously, keeping the gate a single source of truth instead of duplicating it in the button handler.
  - **Known limitation (pre-existing, shared with normally-scheduled multi-step chains):** the saw-work gate waits *strictly* for the Stop (`'waiting'`) signal once a step has been observed busy — it never time-fallbacks in that branch, by design, so a long quiet think is not mistaken for completion. If a step's turn therefore *never* emits Stop (the agent is killed or the terminal is closed mid-chain), the remaining steps hold indefinitely; a multi-step **schedule** in that state won't auto-remove and must be `DEL`-ed manually. Single-step schedules are unaffected (they remove on their first fire). ⚡ NOW reuses the same in-flight machinery, so it shares this edge — it does not introduce a new one.
- **Daily first-fire clamp**: loop items have an optional `firstFireOfDay` (`HH:MM` local time). When set, the scheduler refuses to fire the loop before that clock time on any given day — effectively a positive "start at" version of an exclude window from `00:00` to `HH:MM`. Implemented via `isBeforeDailyStart(item, now)` filtered into `pickNext` and `advanceBlockedLoops`. The meta line shows `— waits until 9:00 AM today —` (italic dim) and the status row appends `(waiting for daily start)`. Schedule items ignore the clamp by design — they have an explicit `runAt`. The clamp IS carried through history snapshots (unlike `disabled`) because it's part of the saved pattern
- **12-hour time picker**: a new `TimePicker12` component (3 dropdowns — Hour 1-12, Minute 0-59, AM/PM) is used everywhere time-of-day entry is required. Replaces native `<input type="time">` in the QueueItemEditModal exclude-window rows, the LoopExcludeWindowsModal session-quiet-hours rows, and the new daily-start-time field. Storage stays `HH:MM` 24-hour everywhere (`db.ts`, snapshots, scheduler) — the picker is purely a render layer. Conversion is one isolated `parseHHMM`/`formatHHMM` pair in `src/lib/timePicker.ts` with full round-trip unit tests covering the midnight (00:00 ↔ 12 AM) and noon (12:00 ↔ 12 PM) edge cases
- **AutocompleteTextarea**: the `/` slash-command / skill / `@` file autocomplete logic was extracted from `QueueTab`'s compose textarea into `src/components/ui/AutocompleteTextarea.tsx`. The same controlled-textarea component now powers four call sites — the queue compose row, the MAIN prompt textarea in `QueueItemEditModal`, every BEFORE chain-step textarea, and every AFTER chain-step textarea. Parent passes `sessionId` (for CLI detect + command-index fetch) and optional `projectPath` (for `@` file search; silently disabled when missing). Pass-through `onKeyDown` lets parents intercept Cmd+Enter and other shortcuts after the autocomplete has had its chance. The dropdown auto-detects direction: expands DOWN if there's ≥220 px below the textarea, UP otherwise — so the compose row at the bottom of the panel still pops upward while the edit-modal textareas in mid-screen pop downward. CSS lives in `src/styles/modules/AutocompleteTextarea.module.css`; the matching `ac*` classes were removed from `Terminal.module.css`. `QueueHistorySheet` plumbs the currently-viewed session's id + project path through when launching the edit modal, so autocomplete stays meaningful when editing a globally-saved history entry
- **Queue-history export / import**: the history sheet header has two icon buttons next to the `×` close — `⬇ Export` and `⬆ Import`. Export dumps every saved entry to `queue-history-YYYY-MM-DD.json` via a Blob + anchor download. Import reads a JSON file via a hidden `<input type="file">`, parses it with `parseImportFile` (in `src/lib/queueHistoryExport.ts`), and shows a preview/confirm modal listing filename + export date + entry count + skipped count. On confirm, the new `bulkImport` action on `queueHistoryStore` writes all rows in a Dexie transaction so a partial failure rolls back. File format is a versioned envelope `{ schema: 'aasc-queue-history', version: 1, exportedAt, count, entries }` — entries omit the DB `id` so re-importing the same file mints fresh ids. Imports always APPEND (never replace); duplicates are kept as-is and the user controls cleanup via the 🗑 row button. Strict validation rejects malformed JSON, wrong schema, newer-version files, and oversized payloads (> 50 MB); per-row rows missing required fields are silently skipped and counted in the preview

## Change Risks
- Auto-send without proper idle detection can send prompts at wrong time
- Session ID migration must update all queue items
- Drag-and-drop reordering must update position field correctly
- **History `historyId` consistency**: edits made via the history sheet do NOT propagate to live queue items — the ★ only marks "this row was saved once," not "this row matches the saved version." Treat as a memory aid, not a sync link
- **Dexie schema bump**: `queueHistory` introduced in v5. New users get it created on first launch; existing users upgrade automatically. Any future field additions to `DbQueueHistory` need a v6 bump or careful migration
- **Snapshot stripping**: `snapshotItem()` in queueHistoryStore drops session-local fields before save. If a new "per-session" field is added to QueueItem, remember to clear it in the snapshot or it will leak into freshly applied items
