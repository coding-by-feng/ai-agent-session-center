# Session Detail Panel

## Function
Slide-in panel showing comprehensive session information across 7 tabs, with a session switcher strip, session control bar, and two mutually-exclusive layout modes (side-by-side split, stacked).

## Purpose
Primary interface for interacting with a single session. Aggregates terminal, project browser, ops commands, conversation transcript, AI-popup history, notes, and prompt queue in one view, plus per-session controls (resume/kill/mute/alert/room) and a horizontal switcher to jump between active sessions.

## Implementation

### Panel shell
- `DetailPanel` wraps everything in `ResizablePanel fullscreen`. The outer overlay is `display:none` (not unmounted) when no session is selected, so `ProjectTabContainer`/terminal state survives deselection.
- `lastSessionRef` keeps the most-recent session as `displaySession` so child containers stay mounted after deselect.
- Minimizable to a draggable badge (`DraggableMiniBadge`), position saved to `localStorage['mini-badge-pos']` (`MINI_BADGE_POS_KEY`). Click restores (only if it wasn't a drag).
- **Maximize / Restore**: SessionSwitcher renders a maximize/restore toggle (`MaximizeIcon`/`RestoreSizeIcon`) backed by uiStore `maximized` (default `false`, **not** persisted; `setMaximized`/`toggleMaximized`). When on, the panel's own session tab strip is hidden — forcing nav back to the slim top bar — to give the terminal more space. (The global Header + NavBar are already hidden whenever the panel is in view — see "Dashboard top-bars auto-hide" below.) `DetailPanel` reads `maximized` and exits maximized mode when the panel closes (no session selected) so reopening always starts un-maximized.
- **Left-docked nav rail**: uiStore `navPosition` (`'top' | 'left'`, persisted to `localStorage['nav-position']`; `setNavPosition`/`toggleNavPosition`) controls where the session nav lives. SessionSwitcher renders a dock-left/dock-top toggle (`DockLeftIcon`/`DockTopIcon`). When `navPosition === 'left' && !maximized`, DetailPanel applies the `panelInnerLeft` class (horizontal `rail | content` layout) and SessionSwitcher renders a vertical rail instead of the top strip; maximize always overrides back to the slim top bar. The rail is `230px` wide (`.switcherBarVertical`). In the narrow rail the control block lays out as clean stacked rows: KILL/MUTE/ALERT on one line, the room `<Select>` (tagged `.roomSelect`, made `flex: 1 1 100%`) dropping to its own full-width line, and the toggle icons spread `space-between` on theirs — instead of all of them wrap-cramming onto shared lines.
- **Fold / expand the left rail**: uiStore `navRailCollapsed` (boolean, persisted to `localStorage['nav-rail-collapsed']`; `setNavRailCollapsed`/`toggleNavRailCollapsed`) folds the left rail to a thin `38px` strip. The toggle only applies to the left dock — `isRailCollapsed = navPosition === 'left' && !maximized && navRailCollapsed`. When **expanded**, a fold button (`CollapseRailIcon`, ◀◀) appears in the rail's control cluster (after maximize, before minimize) — rendered only while vertical, since top-dock has no rail to fold. When **collapsed**, SessionSwitcher early-returns a minimal strip (`.switcherBarVertical.switcherBarCollapsed`, a compound selector that overrides the `230px` width) holding just an expand button (`ExpandRailIcon`, ▶▶) and an active-session count pill (`.railCollapsedCount`, properly pluralized); the right-border divider is preserved and `.mainCol` reclaims the freed width. Collapse is **instant** (no width transition) to avoid xterm `fit` resize thrashing, mirroring how maximize toggles layout. Maximize still overrides to the slim top bar regardless of `navRailCollapsed`.
- **Header title row sizing** (`.switcherToggle` / `.switcherNameDisplay`, shared by top bar + rail): the title (`.switcherNameText`) is the row's focal point — `13px`/`600`/`--text-primary` — beside a `9px` status dot, an `18px` index chip, and the edit/tag/note hint icons (`13px`, `.switcherEditHint`, `opacity: .55` until `.switcherName:hover`). The top bar is `38px` tall / `12px` base font — `48px` when a progress remark wraps onto its own line below. In the narrow rail, `.switcherNameDisplay` is allowed to **wrap** so the `LabelChip` + project name drop to a line below the title instead of crushing it onto a 3–4-character ellipsis.
- Header / title / rename live in **SessionSwitcher**, not DetailPanel — the top row shows a status dot, session-index chip, the title with inline double-click rename (`setSessionTitle`, no separate EditableTitle component) plus its rename/label/note hint icons, the optional progress-remark row beneath, optional project name, status badge, model, duration, the `SessionControlBar` (passed via the `controls` prop), the room filter, the sort-by-recent-activity toggle, then the trailing control cluster in order: status-colour legend, compact/detailed toggle, dock-left/dock-top toggle, maximize/restore toggle, the fold-rail toggle (left-dock only), and the minimize button. In the 230px left rail these eight 26px icons exceed the row's 213px, so `.switcherBarVertical .switcherMeta` sets `column-gap: 4px` to keep all seven toggles on one line; the minimize button wraps below them (as it already did with six toggles). Adding a ninth icon requires re-checking that arithmetic.

### Tabs and mounting
- 7 tabs (`BASE_TABS`): PROJECT, TERMINAL, COMMANDS, CONVERSATION (id `conversation`), AI POPUPS (id `aiPopups`), NOTES, QUEUE.
- TERMINAL, COMMANDS, and PROJECT are **always mounted** (hidden via `display:none`) to preserve the xterm instance/scroll and the file-tree/open-file state. CONVERSATION, AI POPUPS, NOTES, QUEUE mount on demand. Each on-demand tab gets a unique React key (`scroll-<id>`) so scroll position doesn't bleed across tabs.
- The COMMANDS tab is the session's blank "ops" shell (`OpsTerminalContent`); it only renders when `opsTerminalId || hadOpsTerminal`. With no ops terminal it shows a "CONNECT TERMINAL" placeholder that POSTs `/api/sessions/:id/reconnect-ops-terminal`.
- The TERMINAL tab (`TerminalContent`) stacks `TerminalContainer` over an inline `QueueTab` (`bottomRow`), and exposes Reconnect / Fork / Clone actions:
  - Reconnect (`onReconnect`) shown for ended SSH sessions or sessions with a `startupCommand`; POSTs `/api/sessions/:id/reconnect-terminal`.
  - Fork (`onFork`, Claude/Codex only via `detectCli`) POSTs `/api/sessions/:id/fork`; the new `terminalId` is selected and added to the same room.
  - Clone (`onClone`) POSTs `/api/sessions/:id/clone`; same select-and-room behaviour.
  - `TerminalContainer` is passed `originSessionId={sessionId}` so the select-to-translate popup and "Translate previous answer" toolbar action can fork a floating session (see [Floating Terminal Fork](./floating-terminal-fork.md)).

### Layout modes (split / stacked) — mutually exclusive
The PROJECT tab shows two layout toggle icons plus a pop-out-to-window icon (wide panels only, `SPLIT_MIN_WIDTH = 700`). Enabling one layout mode disables the other via the shared `turnOff` helper:
- **Split**: `DraggableSplitView` shows Terminal (left) + Project (right) with a `col-resize` divider, ratio clamp 0.15–0.85. Mode flag `split-terminal-project:{sessionId}` (`SPLIT_KEY`), ratio `split-ratio:{sessionId}` (`SPLIT_RATIO_KEY`, also mirrored to a global fallback).
- **Stacked**: `StackedSplitView` shows Terminal (top) + Project (bottom) with a `row-resize` divider, ratio clamp 0.2–0.8 — fits the ~480px panel better than side-by-side. Mode flag `split-stacked-terminal-project:{sessionId}` (`STACKED_KEY`), ratio `split-stacked-ratio:{sessionId}` (`STACKED_RATIO_KEY`, also mirrored to a global fallback).
- The third icon is **not** a layout mode — it is the stateless pop-out action that opens PROJECT in its own OS window; see [Pop-out to a native window](#pop-out-to-a-native-window).
- `effectiveTab` resolves to `'split'`/`'stacked'` when the corresponding mode is on and the active tab is terminal/project; in those modes the always-mounted terminal/project hosts are suppressed (the combined view renders its own single copies to avoid duplicate xterm subscriptions). Both flags restore per-session on session switch (default off so the previous session's state doesn't leak).

#### Pop-out to a native window
The **TERMINAL** and **COMMANDS** tabs can each be popped out into their own lightweight native OS window (Electron only, draggable to another monitor) via a `⧉` toolbar button. An in-app DOM panel is confined to the app window, so cross-monitor movement requires a real window. Both popouts open a **content-only** window (just that tab's content, not the whole dashboard) loaded from `/?popout=…` — `main.tsx` detects the `popout` query param before bootstrapping `<App>` and renders the popout view directly:

| Tab | Trigger | Window | View |
|-----|---------|--------|------|
| TERMINAL | `⧉` on `TerminalToolbar` (when `onPopOut` provided) | `electronAPI.openTerminalWindow({ terminalId, originSessionId, label })` → `/?popout=terminal&terminalId=…` | `PopoutTerminalView` |
| COMMANDS | `⧉` on `TerminalToolbar` | same, with the **ops** `terminalId`, `label: 'Commands'` | `PopoutTerminalView` |
| PROJECT | **float** icon on the PROJECT tab (wide panels) | `electronAPI.openProjectWindow({ path, label: 'Project' })` → `/?popout=project&path=…` | `PopoutProjectView` (wraps `ProjectBrowserView`) |

**Terminal / Commands popout** (`TerminalContent` / `OpsTerminalContent` in `DetailPanel.tsx`): `handlePopOut()` opens the window and, on success, marks the `terminalId` in `floatingSessionsStore.poppedOut` and calls `electronAPI.unsubscribePty(terminalId)` to release the in-app PTY subscription. While popped out, the tab renders `PoppedOutTerminalPlaceholder` ("…open in a separate window — close it to dock here", FOCUS WINDOW button) instead of the live `TerminalContainer`, so only the popout window subscribes to the PTY (no two-subscriber contention; mirrors the fork-popout trick). **Re-dock** reuses the existing path: closing the window fires `popout:closed` → `FloatingTerminalRoot` calls `setPoppedOut(id, false)` → the in-app `TerminalContainer` re-mounts and replays the PTY buffer. The `⧉` button only renders under Electron (`canPopOutWindow()` = `!!electronAPI?.openTerminalWindow`) and never on the fullscreen-overlay toolbar.

**PROJECT popout** (`DetailTabs.tsx`): the PROJECT tab's **float** icon (shown on wide panels, next to the split/stacked toggles) is a **stateless action** — `openProjectWindow()` calls `electronAPI.openProjectWindow({ path, label: 'Project' })` (native `BrowserWindow` placed on the second monitor via `computePopoutBounds`, bounds persisted, **de-duped by path** so a repeat click focuses the existing window), or falls back to `window.open('/project-browser?path=…', 'aasc-project-<path>')` in the browser. It is gated on the session having a `projectPath`. Unlike the terminal popout there is **no placeholder / re-dock** — the in-app PROJECT tab stays fully usable (the popout is a fresh `ProjectBrowserView`, not the live in-app DOM), and in-app side-by-side is still available via the **split** / **stacked** modes. The previous in-app draggable `FloatingProjectPanel` overlay (and its pop-out gestures) was retired in favour of this — a DOM overlay can't cross monitors; only a real OS window can.

**Known limitations:** the project popout opens at the project root (the float's currently-open file isn't forwarded yet); a select-to-translate fork spawned from inside a popped-out terminal renders in the main window (consistent with the existing fork popout). See [IPC transport](../electron/ipc-transport.md) and [App lifecycle](../electron/app-lifecycle.md) for the native-window side, and [Floating Terminal Fork](./floating-terminal-fork.md) for the shared popout infra.

### Search
- A search bar sits between the tabs and content (`searchOpen` toggles `.hidden`). It searches the session's `promptHistory` text, shows a match counter (`N/total` or `0 results`), and Prev/Next navigation.
- `navigateMatch` auto-switches to the CONVERSATION tab, then scrolls the active `.search-highlight` into view (adding `.search-highlight-active`).
- Opened via the `detail-panel:find` CustomEvent (from the keyboard-shortcut handler); closed on Escape or session change.

### Tab switching via keyboard
- DetailPanel listens for the `detailTabs:switchTab` CustomEvent (dispatched from `useKeyboardShortcuts.ts`) and drives `externalTab`, which DetailTabs consumes via `externalActiveTab`. `externalTab` is cleared ~50ms after set so subsequent user clicks aren't blocked. The same `externalTab` mechanism switches to PROJECT when a terminal file link is clicked (`pendingFileOpen`).
- Active tab persisted in `localStorage['active-tab']` (`STORAGE_KEY`).

### SessionSwitcher details
- Session tab cards use a **neutral static frame** (`var(--border-subtle)`); status is conveyed only by the coloured corner badge (StatusGlyph / green ✓), the mini status dot, and (compact mode) the title colour. The old per-status conic-gradient spinner + pulse border animations were removed as a continuous CPU/GPU drain — do not reintroduce always-running gradient/box-shadow animations on cards (the same removal applies to HeaderAgentStrip `.miniRobot` chips, which keep their static status-coloured border). The `.sessionTabIndex` badge sits at `left: -9px`.
- **Completed badge**: a session that transitions to `waiting` (or, for Codex, jumps `idle`→`waiting`) while not selected gets a pulsing **green `✓` badge** (`attentionIds`, `.sessionTabAttentionBadge`), cleared when switched to. It signals "finished its work, ready for review" — green ✓ rather than a red `!` alarm, since completing is a good state.
- **Per-status glyph badge** (`StatusGlyph` + `.sessionTabStatusBadge`): the top-right corner shows a **distinct icon per status** so states are tellable apart by shape, not colour alone (`waiting`/`prompting` even share cyan): `waiting`→✓, `approval`→`!`, `input`→`?`, `working`→spinner, `prompting`→up-arrow, `idle`→pause-bars, `connecting`→ellipsis, `ended`→✕. The badge is filled with the status colour (`--robot-color`) and carries a dark glyph for contrast. It is **mutually exclusive** with the green ✓ completed badge — the green ✓ wins while `needsAttention`, otherwise the per-status glyph shows. The mapping is **backend-agnostic**: Claude and Codex derive status from the same hook events (`sessionStore` has no per-backend status branching), so the same real-world state shows the same icon on either CLI.
- **Title clearance**: `.sessionTabTitle` carries `padding: 0 18px` (`0 14px` compact) so the absolutely-positioned index badge (top-left) and completed badge (top-right) never overlap the title text.
- **Queue-hint badge** (`.sessionTabQueueBadge` + `QueueBadgeIcon`): when a session has queued prompts, its tab card shows a cyan **list-glyph + count** (e.g. `≣2`) rendered inline at the **start of the title** (an `inline-flex` span inside the `-webkit-box` line-clamp, so the title still wraps/clamps around it and the badge is never pushed out). Hidden entirely when the queue is empty (no `0`). The count comes from the **client `queueStore`** — `useQueueStore(s => s.queues.get(sessionId)?.length ?? 0)`, matching the QueueTab's `QUEUE (N)` header — **not** `session.queueCount`, which is never synced to the client (the `update_queue_count` WS message is defined but unused). Because the tab strip lists the *non-current* sessions, the badge surfaces "which other agents have work waiting" at a glance; the current session's queue shows in its own QueueTab.
- **Per-session label**: a tag button next to the title (in the header) opens `LabelPicker` (built-in ONEOFF/HEAVY/IMPORTANT + custom labels with colors); the chosen label renders as a colored `LabelChip` next to the title in the header and as a small read-only chip/dot on the strip cards. Client-only, stored in [`labelStore`](./state-management.md) (no server).
- **Per-session progress remark**: a free-text note ("what is this session waiting on?") the user maintains by hand, rendered on its own row **under the title**. Unlike the label it is **server-persisted** — `sessionStore.setSessionRemark` optimistically updates the in-memory session and `PUT`s to [`/api/sessions/:id/remark`](../server/api-endpoints.md) → `setSessionRemark` → the `sessions.remark` column ([database](../server/database.md)). Capped at **200 chars on both sides** (client `maxLength`, server `remarkSchema`); it renders on one line and ellipsizes, so the cap must stay in sync at both ends.
  - **Entry point is the note icon** (`NoteIcon`), the third hint button in the title row after rename (`EditIcon`) and label (`TagIcon`). There is deliberately **no empty-state placeholder row**: the row only renders when `currentRemark || remarkEditing`, so the `38px` top bar grows to `48px` only for the sessions that actually carry a remark (an always-on `+ add remark` row taxed every session a second line). The icon gets `.switcherHintActive` — the shared "lit" style also used by the tag button while its picker is open — whenever the session has a remark, so the bar still signals "a note exists" at a glance.
  - **Commit semantics**: Enter and blur (click-away) **save**; Escape **discards**. An empty string is a legitimate edit that *clears* the remark (stored `NULL`), so no truthiness guard may be added on the commit path. Clicking the note icon while the editor is open also saves — the button calls `preventDefault()` on `mousedown` so the input never blurs; without that, blur→commit would flip `remarkEditing` false *before* the click ran and the toggle would immediately re-open the editor it was meant to close.
  - **Rail width trap**: in the left rail `.switcherToggle` is a **wrapping column**, which reverses the meaning of the top-bar `flex: 0 0 100%` on `.switcherRemark` — there it sizes the *main* axis (height), leaving the width to fit the text, so a long remark grew past the rail edge (measured `674px` inside a `230px` rail). `align-self: stretch` does **not** fix it: a wrapping column stretches an item to its flex *line*, whose width is set by its widest item — the overflowing text itself. The `.switcherBarVertical` override therefore pins `flex: 0 0 auto` + `width: 100%`; only a definite width breaks that circle and lets `text-overflow: ellipsis` clip.
- **Room grouping**: sessions sharing a room render inside a room-colored frame (`ROOM_COLOR_PALETTE`, matching HeaderAgentStrip), ordered by `roomIndex`; orphan sessions follow. A multi-select **room filter** dropdown (funnel icon) narrows the strip (`selectedRoomIds`); the current session is never filtered out. Room framing is skipped entirely in activity-sort mode (below).
- **Sort by recent activity** (`SortByActivityIcon`, second toolbar slot, right after the funnel): toggles uiStore's `sessionSortMode` between `'room'` (default) and `'activity'`. In `'activity'` mode `tabRenderItems` returns a flat list of `{type:'session'}` items — no room frames, since a room can only sit in one place while its sessions belong all over a recency-ranked list — ordered by [`sortSessionsByActivity`](../../../src/lib/sessionSort.ts) (pinned first, then `lastActivityAt` descending, tie-breaking title then `sessionId` for a total order). Status is ignored: a stale `working` session sits below a fresh `idle` one. The room **filter** still applies; only the frames vanish. Badge numbers are unaffected — `sessionIndexMap` is always derived from the status sort, so rows reorder without renumbering (which is why the strip's numbers look unsorted in this mode, by design).
- **Status-colour legend**: a separate toolbar button (legend icon) opens the `.statusLegendDropdown` popover mapping each `STATUS_COLORS` swatch to its label under the active theme.
- **Viewport-edge clamping**: both toolbar popovers (`.roomFilterDropdown`, `.statusLegendDropdown`) anchor with `right: 0` and extend leftward, so when the panel is docked left / near the window's left edge they used to clip. [`useDropdownFlipX`](../../../src/hooks/useDropdownFlipX.ts) measures the rendered menu on open and imperatively writes a `translateX(...)` to nudge it back inside the left/right viewport edges (8px pad). It corrects horizontal overflow only — vertical placement stays in CSS, mirroring `Select`'s `flipUp` flip.
- **Collapse / expand a room**: each room frame carries a chevron toggle (`.roomCollapseToggle`) that calls `roomStore.toggleCollapse(roomId)`. When `room.collapsed` is true the frame hides its session cards and shows a count pill (`.roomCollapsedCount`) instead; the chevron rotates to indicate state. The `collapsed` flag lives on the `Room` (persisted to `localStorage['session-rooms']`), so it is shared with — and honoured by — the [HeaderAgentStrip](../3d/robot-system.md) room frames.
- **Room-frame header row** (`.roomHeaderRow`): the collapse chevron (`.roomCollapseToggle`), the kill-all skull (`.roomKillToggle`), and — when collapsed — the count pill (`.roomCollapsedCount`) are wrapped in one flex-row `div` so they stay on the **same horizontal line**. Without it, the vertical rail's `.switcherBarVertical .sessionTabRoomGroup { flex-direction: column }` made each icon its own stacked row (and stretched the collapsed count into a full-width bar). The session cards render as siblings **below** the header row and keep their column stacking in the rail; the horizontal top-bar layout is unchanged (the group is already a row there).
- **Kill all sessions in a room**: each room frame carries a **skull icon** (`KillRoomIcon` + `.roomKillToggle`, dim at rest, red on hover/focus) right after the collapse chevron, inside the `.roomHeaderRow`. It renders only when the room has ≥1 **live** session (`roomLiveCount`, computed by counting `room.sessionIds` whose session exists and `status !== 'ended'`), so a room of only ended cards shows no icon. Clicking it `stopPropagation`s (no session-switch) and calls `uiStore.openRoomKill(roomId)`, which sets `roomKillTargetId` + `activeModal = ROOM_KILL_MODAL_ID` (`'room-kill-confirm'`). The lazy-mounted `RoomKillConfirmModal` then lists the live sessions (capped at 8 + "+N more"), and on confirm `Promise.all`s a per-session kill that mirrors `KillConfirmModal` exactly (`POST /api/sessions/:id/kill` → `DELETE /api/terminals/:terminalId` best-effort), then toasts a summary (`Terminated N…` / `Terminated K of N… — F survived` / `Failed to kill…`) and deselects if the open session was among the killed. **`roomKillTargetId` is the modal's source of truth** — it kills the room's live sessions regardless of the current room filter, and the icon is a single-click-safe affordance (the modal always confirms). Because room frames are skipped in activity-sort mode, the kill icon is absent there too.
- **Display mode toggle** switches between compact (title-only) and detailed (mini robot face + project + CLI badge) cards (`cardDisplayMode` in uiStore). Alongside it, the **dock toggle** (`navPosition`) docks the session nav as a left rail or back on top, and the **maximize toggle** (`maximized`) hides the session strip entirely for more terminal space.
- **CLI badge** (`getCliBadge`) derives CLAUDE/CODEX/GEMINI/AIDER from the SSH command or `backendType`.
- **Recompute gating (perf)**: the `sessions` Map reference changes on every update (`new Map(...)`). To skip the O(N) sort + room-grouping on unchanged content, SessionSwitcher derives a stable `sessionsSignature` from `sessionId|status|pinned|title|projectName|colorIndex|accentColor|terminalId` (sorted + joined) and keys the heavy memos (`sortedSessions`, `activeSessionIds`, and downstream `filteredSessions`/`tabRenderItems`) to that signature instead of the Map reference. **`lastActivityAt` is appended to the signature only when `sessionSortMode === 'activity'`** — it ticks on every hook event, so including it unconditionally would defeat this gating, while omitting it in activity mode would freeze the list (nothing else in the signature changes when a session merely stays busy, so rows would never reorder).

### SessionControlBar
- **RESUME** (only when status `ended`) POSTs `/api/sessions/:id/resume`; inflight fetch is aborted on unmount/session change.
- **KILL** opens `KillConfirmModal` (`openModal(KILL_MODAL_ID)`); the modal POSTs `/api/sessions/:id/kill` `{confirm:true}`, then DELETEs `/api/terminals/:terminalId`, toasts the PID, and deselects.
- **MUTE/UNMUTE** toggles `session.muted` (`toggleMute` + `muteSession`/`unmuteSession` in alarmEngine).
- **ALERT** toggles `session.alerted` (`toggleAlert` + `alertSession`/`unalertSession`) — loud sounds for approval & completion. (Distinct from the dormant `AlertModal`.)
- **Room `<Select>`** moves the session: removes it from any room that currently contains it, then adds it to the chosen room (`''` = No room).

### Modals
- `KillConfirmModal` is lazy-mounted via the `LazyModal` wrapper (only mounts when `activeModal === KILL_MODAL_ID`) to avoid extra Zustand subscriptions during DetailPanel mount.
- `RoomKillConfirmModal` is lazy-mounted the same way (`activeModal === ROOM_KILL_MODAL_ID`). `ROOM_KILL_MODAL_ID` and `roomKillTargetId` live in `uiStore` (not the modal component) so `openRoomKill` can reference the id without a store↔component import cycle; `closeModal` clears both `activeModal` and `roomKillTargetId` so a stale room can't leak into the next open. See the room kill bullet under **SessionSwitcher details**.
- `AlertModal` (timer alert → `db.alerts`) is not currently mounted anywhere; it is documented here as a dormant component.

### Persistence
- Selection: `localStorage['selected-session']`, restored on refresh.
- Nav dock position: `localStorage['nav-position']` (`'top'`/`'left'`), restored on load. Nav-rail folded state: `localStorage['nav-rail-collapsed']` (`'1'`/`'0'`, uiStore `navRailCollapsed`), restored on load (only has a visible effect in left-dock mode). Maximized mode (uiStore `maximized`) is intentionally **not** persisted — it resets to off each load and when the panel closes.
- Visited projects: a Map of every visited sessionId keeps each `ProjectTabContainer` mounted; on session re-key (e.g. after `/clear`, via `replacesId`) the per-session `agent-manager:project-tabs:session:{id}` and `agent-manager:file-tabs:{id}:{tabId}` localStorage keys are migrated from the old id to the new one.
- Close: Escape is a three-step handler — it closes search first, else exits maximized mode, else restores a minimized panel; it does **not** deselect (deselect would apply `display:none` and reset scroll to 0). Escape is ignored when an xterm has focus. The SessionSwitcher "‒" button calls `minimizeDetailPanel` (sets `detailPanelMinimized`), the panel's only non-destructive close gesture; full deselect (`selectedSessionId → null`) happens only on kill (`KillConfirmModal`).
- **Dashboard top-bars auto-hide**: while the panel is in view, `AppLayout` hides **both** the global `<Header>` and `<NavBar>` (`hideTopBars = !!selectedSessionId && !detailPanelMinimized`) so the panel reclaims the full height. Minimizing to a badge or deselecting restores both bars. See [Views & Routing](./views-routing.md) "Top-bars auto-hide".

## Change Risks
- Defining `TerminalContent`/`OpsTerminalContent` inside `DetailPanel` (instead of module scope) remounts the terminal on every render — keep them outside.
- Adding heavy components without lazy mounting (or without the always-mounted-host pattern for terminal/project) degrades performance.
- Changing tab ids/order requires migrating `localStorage['active-tab']` and updating the keyboard-shortcut tab bindings.
- The two layout modes must stay mutually exclusive — a new mode must clear the others via `turnOff`, or duplicate xterm/project subscriptions appear.
- SessionSwitcher's heavy memos depend on `sessionsSignature`, NOT on `sessions`. If you add a card-display field, include it in the signature or the tab strip won't refresh; re-adding `sessions` to the dep array re-enables the every-update recompute this optimization avoids.
- `AlertModal` is dormant — before relying on it, wire an opener (`openModal(ALERT_MODAL_ID)`) and a render site, or remove it.
